# 掲載リンクの鮮度点検

## なぜやるか

ProofLoopの信頼性は「公式・無料の一次情報を主役に置く」という設計で担保されている。ところが一次情報は**静かに腐る**：年度更新で内容が入れ替わり、制度改正で金額が変わり、サイト改編でURLが移転する。エラーも警告も出ないまま、ある日「古い情報を自信満々に載せているサイト」になる。

この点検の目的は、その静かな腐敗を人間が気づける形にすること。**自動で直すことはしない**。判断材料を出して、人が決める。

## 点検対象

2か所に散らばっているので両方見る。

1. **`lib/guide/resources.ts`** … `/guide/living-alone` と `/guide/money` のリソースカード。`kind` が `official` / `guide` / `affiliate`。
2. **Supabase の `job_listings`（`is_active = true`）** … `/baito` の求人カード。`is_affiliate` で広告か公式かが分かれる。URLは `affiliate_url` 列（列名に反して、非広告の公式URLもここに入る）。

## ⚠️ 広告リンクはHTTPで叩かない

バリューコマースの `ck.jp.ap.valuecommerce.com/servlet/referral?...` を curl すると、**それがクリックとして計測される**。成果レポートのEPCが歪むうえ、機械的なアクセスは不正クリックとみなされる恐れがある。点検のために収益と信用を毀損したら本末転倒なので、広告リンクは踏まない。

代わりにこうする：

- **遷移先ドメインを直接叩く**（例：アルバイトEXなら `https://arubaito-ex.jp/`、エネチェンジなら `https://enechange.jp/`）。広告主のサービス自体が生きているかはこれで分かる。
- **プログラム終了・提携終了はVC管理画面でしか分からない**。疑いがあれば `docs/owner-todo.md` に「VCで提携状況を確認」と起票する（管理画面はオーナーのログインが必要）。

## 手順

### 1. 対象リンクを集める

`lib/guide/resources.ts` を読み、Supabase を引く：

```sql
select title, type, is_affiliate, affiliate_url
from job_listings where is_active = true order by display_order;
```

Supabaseのプロジェクトは `uhhofjcyotfyrlhaguvy`（ProofLoop本番）。Glow等の別プロジェクトと取り違えないこと。

### 2. 死活チェック（広告以外）

まとめて叩く。リダイレクトは追い、UAを付ける（付けないと弾くサイトが多い）：

```bash
curl -s -o /dev/null -w "%{http_code}" -L --max-time 15 -A "Mozilla/5.0" "$URL"
```

ステータスの読み方：

| コード | 判定 | 意味 |
|---|---|---|
| 2xx | **ok** | 生きている |
| 401 / 403 / 405 / 429 | **blocked** | bot遮断の可能性が高い。**deadと混同しない**。実在するがcurlを拒否しているだけのことが多い（Indeed、経済産業省など実例あり）。ブラウザで目視して判断する |
| 404 / 410 / 5xx / 000 | **dead** | 要対応。000は名前解決失敗や接続不可＝サービス終了の可能性 |

### 3. 陳腐化判定（`official` かつ 2xx のものだけ）

WebFetchでページを読み、次の観点で見る：

- **年度・年号**が古いまま放置されていないか。逆に更新されていて、ProofLoop側の説明文と食い違っていないか
- **制度・金額・募集要項**が変わっていないか（奨学金の額、申請期限など）
- **ページの主題がラベルとずれていないか**。サイト改編で統合・移転し、リンク先が想定と別物になっていることがある

判定は「**要確認 / 問題なし**」の二値と、理由を1行。**断定できないときは要確認に寄せる**。見逃して古い情報を出し続ける損失の方が、確認の手間より大きい。

`affiliate` と `guide` は中身の判定をしない。広告は遷移先が広告主なので判定に意味がなく、guideは自サイト内の説明。

### 4. 報告する

`docs/link-freshness-report.md` を上書きし、会話でも要点を伝える。テンプレート：

```markdown
# リンク鮮度点検レポート

**実行日：YYYY-MM-DD** ／ 点検 N件（一次情報 N・広告 N・求人サイト N）

## 要対応（dead）
| 掲載箇所 | ラベル | URL | 状態 |
| --- | --- | --- | --- |

## 目視が必要（blocked）
| 掲載箇所 | ラベル | URL | コード |
| --- | --- | --- | --- |

## 内容が古い疑い（stale）
| 掲載箇所 | ラベル | 理由 |
| --- | --- | --- |

## 問題なし
N件（内訳を1行で）
```

該当が0件のセクションは「なし」と書く。**空欄で濁さない**——「見たけれど問題なかった」と「見ていない」は別物で、後から読む人にはその区別が要る。

### 5. 実行日を記録

`docs/.link-freshness-last-run` に実行日を `YYYY-MM-DD` の1行で書く。セッション開始フックがこの日付を読んで、30日を過ぎたらリマインドする。**これを忘れるとリマインダーが止まる**ので必ず書く。

## 直すときの原則

- **URLを黙って書き換えない。** 移転先が見つかっても、提案して承認を得てから直す。掲載リンクは信頼性に直結するので、勝手に張り替えない（CLAUDE.md §5と同じ考え方）。
- **dead を放置しない。** 代替の一次情報が見つからないなら、そのリンクを外すことも選択肢。リンク切れを残す方が損なわれるものが大きい。
- **リンクを直したら `npm run test` を通す。** `resources.ts` には「広告のみのグループを作らない」「広告URLを official ラベルで置かない」を検証するテストがある。リンクを消した結果その不変条件が壊れることがある。
- **広告リンクが怪しいときはVC管理画面**。提携終了・プログラム終了はコードからは分からない。`docs/owner-todo.md` に起票してオーナーに渡す。
