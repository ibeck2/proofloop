## 1. 背景と意思決定の経緯

### 1.1 ターゲット方針

イシューツリーで定義された上位学生（東京一科＋旧帝＋早慶上智ICU）をターゲットとする。ただし既存の一般語ページ（`/baito`・`/guide/money`・`/guide/living-alone` 等）は**併存**させる。

- 既存ページは現に順位を取りに行っている資産であり、上位層向け仮説には実データが存在しないため、既存を捨てる判断はしない。
- 代わりに**上位校比率を推定する計測装置を先に設置**し、実データが出てから配分を決める。
- CLAUDE.md のペルソナ「一人暮らし大学生」は当面書き換えず、上位層向けを別レイヤーとして積む。

### 1.2 トピック選定の根拠（Ahrefs実測・2026-07-21取得）

詳細は `docs/seo/keyword-facts.md`。要点のみ：

| KW | Vol/月 | KD | TP |
|---|---:|---:|---:|
| **gpa 計算** | **2,300** | **0** | **2,000** |
| 大学 gpa | 1,200 | 4 | 400 |
| 長期インターン | 8,400 | 61 | 9,100 |
| 交換留学 | 300 | 0 | 10 |

- 当初は「交換留学」を第一弾候補としたが、実測でクラスタ合計が月600程度と判明し、主軸には小さすぎるため却下。
- 「長期インターン」は KD61 で DR0 の現状では正面突破できないため、権威獲得後に温存。
- **`gpa 計算` は Vol 2,300 / KD 0 / TP 2,000**。この規模で KD0 は明確な市場の穴。

### 1.3 SERP検証（`gpa 計算`・日本・2026-07-21）

上位を占めるのは①汎用計算サイト（keisan.casio.jp）②大学の学内PDF（長崎大・芦屋大・早稲田大）③弱小ツールサイト（ulabo.org はDR17／chu-pick は被リンク0）。**ページ単位の権威（UR）は全て0〜4**で、DR0のドメインでも十分に食い込める。

決定的な発見：**上位の計算ツールはいずれも日本の大学ごとのGPA換算方式に対応していない**。日本の大学は4段階／5段階／素点換算など方式が大学ごとに異なるため、ここが空白地帯であり本施策の差別化の核となる。

副次的な発見：SERPに青山学院大「留学用GPA」・berkeleyhouse「留学に必要なGPA」が入っており、**GPA→留学という検索行動が実在する**。第二弾に交換留学を置く設計の裏付けとなる。

### 1.4 なぜこの施策が計測要件も同時に満たすか

大学ごとに換算方式が異なる以上、**「大学名を選ばせる」UIは機能上の必然**である。したがって 1.1 で必要とした上位校比率の計測装置を、後付けの設問ではなくプロダクトの必須機能として自然に埋め込める。

## 2. 目的とスコープ

### 2.1 目的

1. `gpa 計算`・`大学 gpa` で上位表示を獲得し、上位校学生の流入経路を作る。
2. 流入者の大学分布を計測し、上位校比率の実データを得る（1.1の判定材料）。
3. 上位校からリンクされる資産を1つ作る（イシューツリー章3.3）。
4. GPA帯に応じて第二弾（交換留学）および既存資産へ送客する導線を作る。

### 2.2 スコープに含むもの

- `/gpa` ページ（計算ツール＋SEO本文＋構造化データ）

- `lib/gpa/calculate.ts` の単体テスト（vitest 新規導入）

### 2.3 スコープに含まないもの（明示的に除外）

- 成績データのサーバー保存・ユーザー紐付け

## 3. アーキテクチャ

### 3.1 URL設計

**`/gpa`（トップレベル・新規）**

`/guide/credits/gpa` としない理由：

1. **役割が異なる**。`/guide/credits` は読み物（履修・卒業要件の解説記事）、`/gpa` はツール。検索意図が「読む」と「使う」で別であり、1ページに混ぜると両方の意図充足が中途半端になる。
2. **被リンク資産として短いURLが有利**。学生団体・サークルが貼る際に `proofloop.jp/gpa` は貼りやすい。
3. `/guide` 配下は新入生向けガイドのトーンで統一されており、本施策のターゲット（上位校の在学生・留学出願層）とペルソナがずれる。

**ナビゲーションには追加しない**（CLAUDE.md §5「ナビゲーションには新コンテンツページを追加しない」を遵守）。導線は `/guide/credits` 内・`/guide` ハブ・フッターから確保する。

### 3.2 ファイル構成

| ファイル | 責務 |
|---|---|
| `app/gpa/page.tsx` | `Metadata`、構造化データ、SEO本文（GPAとは／大学で計算方式が違う理由／FAQ）。サーバーコンポーネント |
| `app/gpa/GpaCalculatorClient.tsx` | 計算UI（科目入力・大学選択・結果表示・CTA）。クライアントコンポーネント |
| `lib/gpa/types.ts` | `GradeScale` / `University` / `Course` / 計算結果の型定義 |
| `lib/gpa/scales.ts` | 換算方式マスタ（4段階・5段階・素点換算などのパターン） |
| `lib/gpa/universities.ts` | 上位13校のマスタ（大学 → 換算方式、出典URL、確認日） |
| `lib/gpa/calculate.ts` | GPA計算の純粋関数。UIから完全に独立 |
| `lib/gpa/calculate.test.ts` | `calculate.ts` の単体テスト |

既存コードは `app/guide/credits/page.tsx`（578行）や `app/guide/study-abroad/recommend/StudyAbroadRecommendClient.tsx`（611行）のように、データ定数・ロジック・UIを1ファイルに同居させるパターンを取っている。本施策では**計算ロジックのみを例外的に分離**する。理由は、留学診断が「正解のない推薦」であるのに対し、GPA計算は**正解のある計算**であり、誤りが誤情報として直接ユーザーに届くため。UIと混ぜると単体で検証できない。

その他の部分（SEO本文のデータ定数など）は既存パターンに合わせ、不要な構造化はしない。

### 3.3 データモデル

```ts
// lib/gpa/types.ts

/** 成績評語1つと、それに対応するGP */
export type GradeOption = {
  /** 表示ラベル。例: "秀", "A", "優" */
  label: string;
  /** グレードポイント。例: 4 */
  point: number;
};

/** GPA換算方式 */
export type GradeScale = {
  id: string;
  /** UIに出す方式名。例: "4段階（秀・優・良・可・不可）" */
  label: string;
  /** 'grade' = 評語から換算 / 'score' = 素点から換算 */
  method: "grade" | "score";
  /** method === 'grade' のとき必須 */
  grades?: GradeOption[];
  /** method === 'score' のとき必須。素点(0-100)からGPを返す純粋関数 */
  scoreToPoint?: (score: number) => number;
  /** この方式の満点GPA。例: 4 */
  maxGpa: number;
  /** 方式の補足説明（UIに表示） */
  note?: string;
};

/** 大学 */
export type University = {
  id: string;
  /** 正式名称。例: "東京大学" */
  name: string;
  /** 選択UI用の短縮名。例: "東大" */
  shortName: string;
  /** 計測用の区分。'top' = 対象13校 / 'other' = それ以外 */
  tier: "top" | "other";
  /** 参照する GradeScale の id */
  scaleId: string;
  /** 換算方式の出典（大学公式の履修要項・成績評価規程のURL） */
  sourceUrl: string;
  /** 出典を確認した日付。ISO形式 "YYYY-MM-DD" */
  verifiedAt: string;
  /** 大学固有の注意書き（学部により異なる場合など） */
  note?: string;
};

/** 入力された履修科目1件 */
export type Course = {
  id: string;
  name: string;
  credits: number;
  /** method === 'grade' のとき使用。GradeOption.label と一致 */
  grade?: string;
  /** method === 'score' のとき使用。0-100 */
  score?: number;
};

/** 計算結果 */
export type GpaResult = {
  gpa: number;
  totalCredits: number;
  /** GPAに算入された科目数（不可・0単位を含む） */
  countedCourses: number;
};
```

**`GradeScale` と `University` を分離する設計意図**：日本の大学の換算方式は数パターンに収束し、複数大学が同一方式を共有するケースが多い。大学ごとに式を持つと重複が生じる。分離すれば、将来 50〜100校へ拡張する際の追加コストが `University` の1行のみになる。

### 3.4 データ品質のルール（最重要）

**13校それぞれの正確な換算方式は、設計時点では未確定である。推測で埋めてはならない。**

GPAの計算結果が誤っていれば、それは単なるバグではなく上位校学生に対する誤情報であり、E-E-A-T獲得を狙う施策が逆に信頼を破壊する。したがって以下を設計上の強制ルールとする：

1. `University` の `sourceUrl` と `verifiedAt` は**必須フィールド**とする（TypeScript上でオプショナルにしない）。
2. 出典で裏が取れなかった大学は**マスタに載せない**。計算機の大学選択肢に出さず、汎用方式にフォールバックさせる。
3. **実装の最初のタスクを「13校の公式資料調査と記録」**とし、コード実装より前に置く。
4. ページ上に**出典URLと確認日を表示**する。これ自体が競合（カシオ・ulabo・chu-pick）に対する差別化表示になる。

対象13校：東京大学・京都大学・一橋大学・東京科学大学／北海道大学・東北大学・名古屋大学・大阪大学・九州大学／早稲田大学・慶應義塾大学・上智大学・国際基督教大学（ICU）。

なお学部により方式が異なる大学が存在しうる。その場合は `note` に明記し、UI上でも注意書きを表示する。学部別マスタは持たない（YAGNI／保守負荷）。

### 3.5 計算ロジック

```
GPA = Σ(科目のGP × 単位数) ÷ Σ(単位数)
```

- `method: 'grade'` は `GradeOption.point` を、`method: 'score'` は `scoreToPoint(score)` を GP として使う。
- 不可（GP=0）の科目も分母の単位数に算入する（日本の大学の一般的な扱い）。ただし大学によって「不可を算入しない」方式があるため、該当する場合は `GradeScale` を別方式として定義する。
- 履修中・評価対象外（認定単位等）は入力させない。UI上で「GPAに算入される科目のみ入力」と明示する。
- 小数点以下は第2位まで表示（切り捨てではなく四捨五入）。

### 3.6 エラー処理

既存依存の `react-hook-form` + `zod`（導入済み）を使用し、新規依存を増やさない。

| 条件 | 挙動 |
|---|---|
| 科目0件で計算実行 | 「科目を1つ以上入力してください」をインライン表示。計算しない |
| 単位数が0以下または未入力 | 該当行にインラインエラー。計算しない |
| 単位数が非数値 | 同上 |
| 素点が0-100の範囲外 | 該当行にインラインエラー |
| 全科目の単位数合計が0 | 「単位数の合計が0のため計算できません」を表示（ゼロ除算防止） |
| 大学未選択 | エラーとしない。汎用方式で計算し、`university_tier: 'unset'` として計測 |

## 4. 計測設計

### 4.1 方針：GA4イベントのみ。Supabaseテーブルは作らない

必要なのは「どの大学から来た人がどれだけいるか」という**集計値**のみであり、個人の成績を保存する必要が一切ない。保存すれば成績という機微情報を抱えるリスクと、スキーマ変更・RLS設計の承認コスト（CLAUDE.md §5）が発生する。GA4 の tier 別イベント数比較で目的を完全に達成できる。

### 4.2 イベント定義

```
event name: gpa_calculate
params:
  university_id    : string  // 例: "u-tokyo" / "other" / "unset"
  university_tier  : string  // "top" | "other" | "unset"
  gpa_band         : string  // "~2.0" | "2.0-2.5" | "2.5-3.0" | "3.0-3.5" | "3.5~"
  course_count     : number  // 入力科目数
```

計算実行時に1回送信する。成績の生データ・科目名は**送信しない**。

### 4.3 導き出す指標

- **上位校比率** = `university_tier = 'top'` のイベント数 ÷ 全 `gpa_calculate` イベント数
- 大学別の流入分布（`university_id` 別）
- GPA帯分布（第二弾＝交換留学の需要規模の推定に使う）

## 5. 導線設計（CV）

計算結果のGPA帯に応じて次アクションを出し分ける。

| 条件 | 提示する次アクション | 遷移先 |
|---|---|---|
| GPA 3.0以上 | 交換留学の出願要件を満たしているか確認する | `/guide/study-abroad` ＋ 留学診断 `/guide/study-abroad/recommend` |
| GPA 2.0未満 | GPAの上げ方・履修設計を見る | `/guide/credits` |

GPA帯での出し分けは、そのまま第二弾（交換留学）への送客装置として機能する。

## 8. テスト方針

### 8.1 vitest の新規導入

本リポジトリにはテストフレームワークが存在しない。`vitest` を devDependency として追加する（追加は1パッケージのみ）。

- 対象は `lib/gpa/calculate.ts` の単体テストのみ。**既存コードのテストは書かない**（スコープ外）。
- `lib/gpa/calculate.ts` は依存のない純粋なTypeScript関数であるため、React用のテスト環境（jsdom / testing-library）は不要。
- `package.json` に `"test": "vitest run"` を追加する。

依存追加は CLAUDE.md §0 により承認事項であり、本設計書をもって承認済みとする。

### 8.2 テストすべきケース

| ケース | 期待 |
|---|---|
| 4段階方式・複数科目の加重平均 | 手計算値と一致 |
| 単位数が科目ごとに異なる場合の加重 | 単位数の大きい科目の影響が大きいこと |
| 不可（GP=0）を含む場合 | 分母に単位数が算入される |
| 素点換算方式 | `scoreToPoint` の結果でGPが決まる |
| 単位数合計0 | ゼロ除算せずエラーを返す |
| 科目0件 | エラーを返す |
| 小数点処理 | 第2位まで四捨五入 |
