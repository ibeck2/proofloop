# ProofLoop アフィリエイト導入 設計書

**作成日：2026-07-24**
**前提：バリューコマース（sid: 3766669）のURL変更審査が承認済み（2026-07-24）。ProofLoopからのアフィリエイト実装が解禁された。現状、サイトからの収益はゼロ。**

---

## 1. 目的

B2Cコンテンツ面（`/baito`・`/guide` 系）から収益を立ち上げる。ただし**サイトの信頼性を最優先**し、「広告だらけで信用を落とす」状態を構造的に避ける。

---

## 2. 大原則（信頼性ファースト）

この5点を、設計・実装・運用のすべてで守る。コンポーネント側で機械的に強制できるものは強制する。

1. **一次情報（公式・無料）が主役**。広告は選択肢の1つとして横に添える。
2. **全広告リンクに開示**：「※広告」ラベル ＋ `rel="sponsored noopener noreferrer"` ＋ `target="_blank"`。**ステマ規制（景品表示法・2023/10〜）の順守は法的義務**。既存 `/baito` の開示文「※外部サービスに移動します（広告）」を踏襲・共通化する。
3. **不変条件**：各リソースカード群には必ず1つ以上の**非アフィリエイト一次情報**を含める。広告リンクのみのブロックを作らない。コンポーネント側でバリデートする。
4. **就活系NG**：ProofLoopの母体 iBECK が就活領域の事業者であるため、**新卒スカウト・逆求人・転職エージェント系（例：ビズリーチ / UTBoard / OfferBox / dodaキャンパス 等）への直リンクは禁止**。一方、**バイト・インターン・クラウドソーシング（アルバイトEX / クラウドワークス 等）は可**。除外方針は `lib` にコメント付きで定義し、レビューで担保する。
5. **VCアカウント内の非ProofLoopサイトには一切触らない**。ブラウザ操作は sid:3766669 の範囲のみ。

## 4. レイアウト単位：リソースカード方式

テーマ小見出しの下に「一次情報リンク（公式・無料）」と「関連サービス（広告）」を並べたカード。一次情報が主役、広告は選択肢の1つ。

```
□ 引越しを安くする
 ・国民生活センター「引越しトラブル」（公式・無料）
 ・相場の調べ方（解説）
 ・引越し見積りを比べる ※広告  [→]

□ ネット回線を選ぶ
 ・学生向けプランの見方（解説）
 ・回線を探す ※広告  [→]
```

### コンポーネント設計

- `components/guide/ResourceGroup.tsx` … カード群のコンテナ。見出し＋ `ResourceItem` の配列を受け取る。
  - **不変条件を強制**：配下に非アフィリエイト項目が0件ならビルド時/実行時に警告（原則3）。
- `components/guide/ResourceItem.tsx` … 1リンク。`kind: 'official' | 'guide' | 'affiliate'` を持つ。
  - `kind==='affiliate'` のとき、開示ラベル・`rel="sponsored noopener noreferrer"`・`target="_blank"` を**自動付与**（原則2）。書き手が付け忘れられない構造にする。
- UIはタスクA（UI刷新・完了済み）の6色トークンに準拠。深紅の予算（静止1画面2箇所）を超えない。

## 5. リンク管理（データの置き場所）

| 対象 | 置き場所 | 理由 |
| --- | --- | --- |
| guide系 | version-controlled な `lib/guide/resources.ts`（型付きデータ） | git diff で「勝手に張り替えない」を担保（CLAUDE.md §5）。「URLはハードコードせず一箇所で定義」とも整合。管理画面不要 |
| baito | 既存 `job_listings.affiliate_url`（`/admin/jobs` か SQL で投入） | 器が完成済み。求人はスコア・タグを持つ動的データのためDBが適切 |

- guide の全リンク（一次情報＋広告）を `lib/guide/resources.ts` に集約し、**鮮度監視（Phase 5）の単一走査元**にする。

## 6. ワークフロー（PDCAループ）

### Phase 0 — 案件棚卸し（`claude-in-chrome`）★要ログイン済みVCブラウザ
- ユーザーが ValueCommerce にログイン済みのブラウザタブを用意。Claude が sid:3766669 の**承認済み広告主（提携済みプログラム）を列挙**。
- 各ページのテーマに合う広告主を選定（**就活系を除外**、原則4）。MyLink（`ck.jp.ap.valuecommerce.com/...` 形式）のリンクを生成。
- **他サイト（VCアカウント内の非ProofLoopサイト）には触らない**（原則5）。

### Phase 1 — コンテンツ設計
- 各テーマの**一次情報ソース（公式・無料）を先に確定**する。Claude が候補を調査・提案（例：引越し＝国民生活センター、お金＝金融庁／日本学生支援機構）。
- 広告はその一次情報の横に配置。カードごとに不変条件（原則3）を満たすか確認。

### Phase 2 — 実装
- `ResourceGroup` / `ResourceItem` コンポーネント新設 ＋ `lib/guide/resources.ts` ＋ 各ページへ差し込み。
- `/baito` は `job_listings.affiliate_url` に生成済みリンクを投入。
- 開示・rel・target をコンポーネントで共通化。

### Phase 3 — 計測
- **GA4 アウトバウンドクリックイベント**（`affiliate_click`：`page` / `advertiser` / `position` をパラメータに）を各広告リンクに付与。
- VC管理画面の成果レポート（クリック・成果・EPC）と突き合わせ。

### Phase 4 — 改善
- クリック率・成果を見て、カード順・文言・広告主を入れ替え（**すべて git 経由**、原則5）。

### Phase 5 — 一次情報の鮮度監視（月次・自動）
- **実行基盤**：GitHub Actions の `schedule` cron（月1回）。
- **対象**：`lib/guide/resources.ts` の全リンク。
- **チェック（2段）**：
  1. **死活チェック**（無料）：各URLのHTTPステータス（4xx/5xx/リダイレクト先変化）を確認。
  2. **内容変化チェック**（Claude API）：一次情報ページを取得し、Claude が「年度・制度が変わって古くなっていないか」を判定（例：「2025年度の額に更新されている」「募集要項が差し替わっている」）。リンク数×月1回なのでAPIコストは軽微。
- **出力**：**自動修正はしない**。検出結果を **GitHub Issue（または PR ドラフト）に「要確認リンク一覧＋理由」として起票** → 人が判断して更新。ProofLoopの「自動公開しない」原則（CLAUDE.md §7・content-pipeline設計）と整合。

## 7. コンプライアンス

- **ステマ規制（景表法）**：広告であることの明示は法的義務。全アフィリエイトリンクに開示ラベル＋ `rel="sponsored"`。
- **就活競合の除外**：`lib` に除外方針（新卒スカウト・逆求人・転職エージェント系）をコメントで明記し、コードレビューで担保。
- `rel="sponsored noopener noreferrer"` を全広告リンクで統一。
