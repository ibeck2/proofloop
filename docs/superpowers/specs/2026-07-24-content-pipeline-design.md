# AI記事＋SNS生成ループ 設計

**作成日：2026-07-24**
**対象：GitHub Actions × Claude API による、記事とSNS投稿の下書き生成 → 人の承認 → 公開 → 計測 → リライトのループ**

関連：`CLAUDE.md` §6・§7（Phase 1）、`docs/task-board.md` タスクB

---

## 1. 最初に押さえるべき結論

**コストは制約にならない。制約は人間のレビュー時間である。**

後述の見積もりのとおり、記事1本の生成にかかるAPI料金は**40円前後**。週1本なら月160円、週10本でも月1,600円程度で、判断に影響する額ではない。

一方、記事1本を人が読んで事実確認し、公開判断するには**30〜60分**かかる。つまり：

- **「たくさん生成する」ことに価値はない。** 生成量を増やしても、レビュー待ちの下書きが積み上がるだけになる
- **設計の目的は「レビューを速くすること」**に置く。事実の出典を機械側で必ず添える、人が直すべき箇所を機械側で明示する、といった設計がコスト削減より効く
- したがって**モデルは最上位を使う**。安いモデルで品質を落とすと、浮くのは数十円で、増えるのは人間の修正時間である

この判断を最初に固定しておかないと、「安いモデルで大量生成」という誤った最適化に流れる。

## 2. パイプラインの全体像

```
[1] キーワード選定 ──── 人が承認 ──┐
        │ GSC実データ ＋ Ahrefs（登録後）
        ↓                          │
[2] 構成案の生成（Claude）          │
        ↓                          │
[3] 本文の生成（Claude）            │  ← ここまで GitHub Actions が自動
        ↓                          │
[4] 事実チェック（Claude・別プロセス）│
        ↓                          │
[5] SNS投稿の派生生成（Claude）     │
        ↓                          │
   Pull Request として出力 ─────────┘
        ↓
[6] 人がレビュー・修正・マージ  ← ★ここが律速。ここを速くする設計にする
        ↓
[7] 公開（Vercel）
        ↓
[8] 計測（GSC / GA4 / SNS）── 2週間後
        ↓
[9] リライト判定 → [2] へ戻る
```

**単発の自動投稿にはしない**（CLAUDE.md §7）。[6]の人間の承認を必ず挟む。SNS投稿も自動投稿せず、下書きとして出す。

## 3. 各段の設計

### [1] キーワード選定

- **入力**：GSCの実データ（Ahrefs MCP経由で今すぐ取得可能）＋ `docs/seo/keyword-facts.md` の既存調査
- **出力**：`docs/seo/queue/<slug>.md`（狙うキーワード・検索意図・想定読者・既存記事との重複チェック結果）
- **自動化しない。** 何を書くかは事業判断なので、候補の提示までを機械が行い、選定は人がする

### [2] 構成案の生成

- 見出し構成、各見出しで答えるべき問い、必要な出典の種類を出す
- **既存記事（`/guide/credits` 等）の構成を参照させる**ことで、トーンと粒度を揃える
- 出力は**構造化する**（後述の Structured Outputs）。自由テキストだと後段で壊れる

### [3] 本文の生成

- 構成案＋文体ガイド＋既存記事2本を入力に、本文を生成
- **文体ガイドと既存記事はプロンプトキャッシュに載せる**（毎回同じ＝キャッシュが効く）

### [4] 事実チェック（別プロセスで実行する）

**同じ文脈で自己チェックさせない。** 書いた本人に検算させても甘くなる。本文だけを渡した別の呼び出しで、次を出させる：

- 事実を述べている箇所の一覧
- それぞれの出典の有無（無いものは「要出典」として明示）
- 断定しすぎている箇所

これは `/gpa` の東大対応で実際に効いた形（推論と事実を分け、推論には注記を付ける）と同じ考え方。**この段の出力が、[6]のレビューを速くする本体である。**

### [5] SNS投稿の派生生成

- 記事から X（Twitter）と Instagram の投稿案を生成
- 診断系（`/baito/simulator`・留学診断・`/gpa`）を拡散フックとして紐づける
- **投稿はしない。** 下書きとして出し、人が投稿する

### [6] 人のレビュー

- 出力先は **GitHub の Pull Request**。理由：差分が見える、コメントで指摘できる、承認とマージが1つの操作、履歴が残る
- PR本文に「要出典の箇所」「断定しすぎの箇所」を[4]の結果として貼る
- 管理画面を作らない。**作る手間に対して得るものが少ない**

### [8][9] 計測とリライト

- 公開2週間後にGSCで順位・表示回数・CTRを取得
- CTRが想定より低ければタイトルとディスクリプションのリライト、順位が低ければ本文の追補
- **リライト判定も自動実行せず、候補の提示までにする**

## 4. 技術選定

### 実行基盤：GitHub Actions

理由：リポジトリにPRを作る処理と相性が良い／スケジュール実行ができる／サーバーを持たなくてよい。

### SDK：`@anthropic-ai/sdk`（TypeScript）

このリポジトリはTypeScriptなので、公式のTypeScript SDKを使う。`zod` は既に依存にあるので、構造化出力にそのまま使える。

### モデル選定

| 段 | モデル | 理由 |
| --- | --- | --- |
| [2] 構成案 | `claude-opus-4-8` | 検索意図の読み取りと構成の設計は、この工程で最も品質差が出る |
| [3] 本文 | `claude-opus-4-8` | 同上。ここでの品質低下が人間の修正時間に直結する |
| [4] 事実チェック | `claude-opus-4-8` | 見落としが最も高くつく工程。安いモデルにしない |
| [5] SNS投稿 | `claude-opus-4-8` | 短文だが、拡散するかどうかは文言の質で決まる |

**全段で最上位モデルを使う。** §1のとおり、モデルを下げて浮く額（記事1本あたり十数円）より、品質低下で増える人間の時間のほうが高い。

`claude-opus-4-8` は入力 $5 / 出力 $25（100万トークンあたり）。参考：`claude-sonnet-5` は $3/$15（2026-08-31まで導入価格 $2/$10）、`claude-haiku-4-5` は $1/$5。

### 主要なパラメータ

```ts
const response = await client.messages.create({
  model: "claude-opus-4-8",
  max_tokens: 16000,
  thinking: { type: "adaptive" },        // 4.8では明示しないと思考しない
  output_config: { effort: "high" },     // 構成・事実チェックは high 以上
  system: [
    { type: "text", text: STYLE_GUIDE, cache_control: { type: "ephemeral" } },
  ],
  messages: [...],
});
```

**注意点（間違えやすい）**

- `thinking` は **`{ type: "adaptive" }`**。`budget_tokens` は Opus 4.8 では**400エラー**になる
- `temperature` / `top_p` / `top_k` は Opus 4.8 では**受け付けられない**（400）。文体の揺らぎはプロンプトで指示する
- **`thinking` を省略すると思考せずに動く。** 明示すること
- `effort` は `output_config` の中。トップレベルではない
- **アシスタントの事前入力（prefill）は400になる。** 出力形式の固定は Structured Outputs で行う

### 構造化出力

段と段の受け渡しは必ずスキーマで固定する。自由テキストで渡すと後段が壊れる。

```ts
import { zodOutputFormat } from "@anthropic-ai/sdk/helpers/zod";
import { z } from "zod";

const OutlineSchema = z.object({
  title: z.string(),
  metaDescription: z.string(),
  sections: z.array(z.object({
    heading: z.string(),
    question: z.string(),        // この見出しが答える問い
    needsSource: z.boolean(),    // 出典が要るか
  })),
});

const res = await client.messages.parse({
  model: "claude-opus-4-8",
  max_tokens: 16000,
  output_config: { format: zodOutputFormat(OutlineSchema) },
  messages: [...],
});
res.parsed_output!.sections;
```

### プロンプトキャッシュ

文体ガイドと参照記事は毎回同じなので、system の最後のブロックに `cache_control` を置く。

⚠️ **キャッシュは前方一致。** system プロンプトに日付や記事IDを差し込むと、以降が全て無効になる。可変の情報は必ず messages 側の後方に置く。

⚠️ **Opus 4.8 のキャッシュ最小長は4,096トークン。** 文体ガイドがそれ未満だと、指定しても黙ってキャッシュされない。`usage.cache_read_input_tokens` が0のままなら効いていない。

### バッチ処理は「使わない」

Message Batches API は**50%安い**が、このパイプラインでは使わない。理由：

- 週数本の生成に対し、割引額は月あたり数十円。**得るものが小さい**
- 完了まで最大24時間かかる。段と段が直列に依存する（構成→本文→チェック）ので、待ち時間が積み上がる
- 実装が複雑になる（ポーリング、結果の突き合わせ）

**将来、既存記事を数十本まとめてリライト判定にかけるときには使う。**その時が来たら導入する。

## 5. コスト見積もり

記事1本あたりのトークン量（既存のガイド記事の規模から推定）：

| 段 | 入力 | 出力 |
| --- | --- | --- |
| 構成案 | 20,000（文体ガイド＋既存記事2本＋KWデータ） | 1,500 |
| 本文 | 22,000（上記＋構成案） | 7,000 |
| 事実チェック | 8,000（本文のみ） | 2,000 |
| SNS投稿 | 9,000（本文＋診断ページの説明） | 800 |
| **合計** | **59,000** | **11,300** |

`claude-opus-4-8`（入力$5 / 出力$25 per 1M）で計算：

- 入力：59,000 × $5 / 1,000,000 = **$0.295**
- 出力：11,300 × $25 / 1,000,000 = **$0.283**
- **合計 約$0.58 ≒ 87円/本**

プロンプトキャッシュを効かせると、2本目以降は入力の大半（文体ガイド＋既存記事＝約18,000トークン）が**キャッシュ読み出し（約1/10）**になる：

- 入力：(59,000 − 18,000×3) × $5/1M + 18,000×3 × $0.5/1M ≒ $0.052
- **合計 約$0.34 ≒ 50円/本**

| 発行ペース | 月間コスト |
| --- | --- |
| 週1本 | **約200円** |
| 週3本 | **約600円** |
| 週10本 | **約2,000円** |

**どのペースでも事業判断に影響しない額。** 繰り返すが、制約はレビュー時間である。

⚠️ この見積もりは**推定トークン量に基づく**。実装したら `client.messages.countTokens()` で実測して置き換えること。特に日本語は英語よりトークン効率が悪い。

## 6. 実装しない判断

| やらないこと | 理由 |
| --- | --- |
| SNSへの自動投稿 | CLAUDE.md §7 の方針。誤爆の損失が大きく、得るものは投稿の手間だけ |
| 記事の自動公開 | 事実誤認が本番に出る。PRの承認を必ず挟む |
| 管理画面の構築 | GitHubのPRで足りる。作る手間に見合わない |
| Batches API | §4のとおり。将来の一括リライト時に導入 |
| 安いモデルへの切り替え | §1のとおり。浮く額より人間の時間が高い |

## 7. 着手順と前提

**先に決まっていないと動けないもの**

1. **記事の置き場所** — `/guide` 配下に足すのか、記事用のディレクトリを新設するのか。**ナビゲーションには追加しない**（CLAUDE.md §5）
2. **公開の承認を誰が行うか** — PRのレビュー担当
3. **Ahrefs の登録**（CEO対応・未完） — 無くてもGSCデータで始められるが、競合の強弱が見えない

**着手順**

1. 文体ガイドを作る（既存の `/guide/credits`・`/guide/money` から抽出する）
2. [2]構成案の生成だけを実装し、手元で回して品質を見る
3. [3]本文、[4]事実チェックを足す
4. PR自動作成を足す
5. [5]SNS投稿を足す
6. 2週間運用してから[8]計測・[9]リライト判定を足す

**最初から全部作らない。** [2]の出力品質が悪ければ、後段を作っても意味がない。
